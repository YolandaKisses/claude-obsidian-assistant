## 31.02 Codex 开发提示词

这份笔记保留我最常用的一套分阶段模板，默认工作流是：

`PRD -> Spec -> implementation plan -> 开发 -> review / 发布前检查`

默认适用场景：
- 单人开发为主
- 常用插件 / skills：`Superpowers`、`Build Web Apps`、`GitHub`、`Google Drive`、`Computer Use`、`CodeRabbit`、`Canva`

常用插件分工：
- `Superpowers`：拆需求、写 PRD / spec、出 plan、调试、验证
- `Build Web Apps`：Web 项目实现，尤其是前端、接口、部署相关
- `GitHub`：读取仓库、issue、PR、历史改动上下文
- `Google Drive`：读取和回写 PRD、spec、会议纪要、表格、Google Docs
- `Computer Use`：走真实网页流程、手工验收、复现问题
- `CodeRabbit`：代码评审、找风险和测试缺口
- `Canva`：快速视觉稿、界面草图、简单设计表达

Google Drive 用法：
- `PRD` 阶段：先找已有需求文档、会议纪要、表格、参考资料
- `Spec` 阶段：补齐边界、状态、口径、约束
- `implementation plan` 阶段：引用确认过的文档作为计划依据
- `开发后`：需要时把 PRD、spec、发布说明写回 Google Docs

---

## 1. 分阶段模板

### 1.1 先进入 PRD 阶段
适合先定方向、范围、目标用户和成功标准，不急着写代码。

```text
先进入 PRD 阶段，不要直接写代码，也不要先给 implementation plan。

请主动结合这些插件 / skills：
- Superpowers：帮我收敛范围、澄清目标、输出 PRD 结构
- Google Drive：如果我提供了文档线索，先读取相关 Google Docs / 表格 / 会议纪要再整理 PRD
- GitHub：如果这是基于现有仓库的项目，读取仓库上下文辅助理解现状
- Canva：如果适合，可以顺手给出页面草图或信息结构建议

等我确认 PRD 后，再上传到 Google Docs。

请先完成：
1. 输出 PRD 草案
2. 标出待确认问题
3. 在我确认前，不要进入 spec、implementation plan 或开发

PRD 需要包括：
- 背景
- 目标
- 非目标
- 目标用户
- 用户场景
- 核心流程
- 范围
- 边界条件
- 验收标准
- 风险点
- 待确认问题

需求：
[你的需求]

```

### 1.2 PRD 确认后进入 Spec 阶段
适合把功能定义、边界、状态和验收口径说清楚。

```text
PRD 我确认了。
现在进入 spec 阶段，不要开始开发，也不要先给 implementation plan。

请主动结合这些插件 / skills：
- Superpowers：把 PRD 细化成可确认的 spec
- Google Drive：继续参考已确认的 PRD、补充文档、会议纪要，并在我确认后写回 Google Docs
- GitHub：如果是现有系统，读取当前实现帮助定义真实边界
- Canva：如果这个功能涉及页面或交互，给出简洁的 UI / 交互说明

等我确认 spec 后，再上传到 Google Docs。

请按下面阶段推进：
1. 基于 PRD 补齐功能定义
2. 分块输出 spec 并逐块让我确认
3. spec 需要写清行为规则、边界条件、状态变化、异常处理、验收口径
4. spec 全部确认后，再单独输出 implementation plan

在我确认前，不要写代码。

PRD 摘要：
[贴 PRD 或 PRD 链接]

```

### 1.3 Spec 确认后进入 implementation plan
适合把需求转成可执行开发计划。

```text
spec 已确认。
请把它转成一份可执行 implementation plan，不要直接开始编码。

请主动结合这些插件 / skills：
- Superpowers：按阶段拆 plan，控制任务粒度和验证方式
- Google Drive：引用已确认的 PRD / spec，等我确认后写回 Google Docs
- GitHub：结合仓库结构、已有模块和 issue / PR 上下文来拆任务
- Build Web Apps：如果是 Web 项目，按实际前后端实现方式拆分任务

请满足这些要求：
- 按阶段拆分
- 每个任务尽量小而清晰
- 标出涉及模块 / 文件
- 写清验证方式
- 标出需要我拍板的决策项
- 需要时引用 Google Drive 中已经确认的 PRD / spec

先输出计划，不要实现。
等我确认 implementation plan 后，再上传到 Google Docs。

```

### 1.4 implementation plan 确认后开始开发
适合真正进入编码阶段。

```text
implementation plan 我确认了。
现在可以开始开发。

请主动结合这些插件 / skills：
- Build Web Apps：负责 Web 项目实现
- Superpowers：按 TDD、拆步实现、持续验证
- GitHub：读取仓库上下文、历史改动和 issue / PR
- Google Drive：把已确认的 PRD、spec、implementation plan 当作实现依据
- Computer Use：如果需要真实走页面流程，就用它辅助验收

请严格按 TDD 执行：
1. 先写失败测试
2. 运行并确认测试失败
3. 写最小实现使测试通过
4. 必要时再重构

每完成一小步就说明你做了什么、改了哪些文件、以及怎么验证的。
```

### 1.5 系统化调试 / 修 Bug
适合 bug 排查，不想让它直接乱改。

```text
帮我系统化排查这个问题。
先不要直接改代码，先做：
1. 复述问题和影响范围
2. 列出可能原因
3. 设计最小复现 / 验证方法
4. 定位根因
5. 给出修复方案
6. 最后再实施并验证

请主动结合这些插件 / skills：
- Superpowers：系统化调试和验证流程
- GitHub：读取相关 issue、历史改动、PR 上下文
- Build Web Apps：如果是 Web 问题，结合实现上下文分析
- Computer Use：如果需要真实点页面复现流程或手动验收
- Google Drive：如果有会议纪要、需求说明或 QA 记录，先吸收
- CodeRabbit：修复后做一轮 review，检查回归风险

问题描述：
[你的 bug 描述]

```

### 1.6 代码评审 / Review
适合让它像 reviewer 一样工作。

```text
帮我做一次 code review。

请主动结合这些插件 / skills：
- CodeRabbit：从 reviewer 视角找 bug 风险、回归风险和测试缺口
- GitHub：结合当前改动、PR、issue、历史实现
- Superpowers：补充系统性验证视角

重点关注：
- bug 风险
- 行为回归
- 边界条件
- 缺失测试
- 不合理的复杂度

请先给 findings，按严重程度排序，再给简短总结。

```

### 1.7 收尾 / 提交前检查
适合功能做完后的收官。

```text
这个功能已经实现完了。
请帮我做上线前检查。

请主动结合这些插件 / skills：
- Superpowers：做完成前检查和风险梳理
- CodeRabbit：补一轮代码风险检查
- GitHub：结合本次改动、PR、issue 和历史上下文
- Computer Use：如果有必要，走一遍真实用户主流程
- Google Drive：需要时把发布说明、验收记录写回 Google Docs

请重点检查：
- 是否和原 spec 一致
- 测试是否充分
- 是否有明显回归风险
- 是否还有边界场景没覆盖
- 是否可以进入提测

请先列问题，再给结论。

```
