
### 1. 进入需求 / Spec 阶段
适合先澄清需求，不急着写代码。

```text
先不要写代码，先进入 spec 阶段。
我会给你一个需求，请你先帮我澄清问题、识别目标用户、梳理范围和边界，并把 spec 按块拆开给我确认。
在我明确确认 spec 之前，不要进入开发，也不要输出 implementation plan。

需求是：
[把你的需求写在这里]
```

更强约束版：

```text
先不要开发，先把这个需求定义清楚。
请按下面阶段推进：
1. 澄清需求
2. 分块输出 spec 并逐块让我确认
3. spec 全部确认后，再单独输出 implementation plan
在我确认前，不要写代码。

需求：
[你的需求]
```

### 2. 生成 PRD / 产品文档
适合先产出产品文档，而不是技术方案。

```text
进入 PRD 阶段。根据下面这次迭代需求写一版 PRD。
先不要开发，也不要给 implementation plan。
PRD 需要包括：背景、目标、非目标、用户场景、核心流程、范围、边界条件、验收标准、风险点、待确认问题。
需求如下：
[你的需求]
```

如果你要配合 Google Docs：

```text
进入 PRD 阶段。根据下面需求写一版 PRD，并放到 Google Docs。
没有我的确认，不要开始开发。
先给我 PRD 草案结构，确认后再写入文档。

需求：
[你的需求]
```

### 3. Spec 确认后进入实施计划
适合把需求转成可执行开发计划。

```text
上面的 spec 我确认了。
现在请单独输出 implementation plan，不要直接开始编码。
要求把任务拆小，明确每一步的目标、涉及文件、前后端改动点、验证方式、依赖关系和风险。
```

更工程化一点：

```text
spec 已确认。
请把它转成一份可执行 implementation plan：
- 按阶段拆分
- 每个任务尽量小而清晰
- 标出涉及模块 / 文件
- 写清验证方式
- 标出需要我拍板的决策项
先输出计划，不要实现。
```

### 4. 开始开发，但要求按 TDD
适合真正进入编码阶段。

```text
现在可以开始开发。
请严格按 TDD 执行：
1. 先写失败测试
2. 运行并确认测试失败
3. 写最小实现使测试通过
4. 必要时再重构
每完成一小步就说明你做了什么和怎么验证的。
```

更严格版：

```text
开始实现，但不要跳过测试。
请按 red -> green -> refactor 执行，不允许先写功能代码再补测试。
如果发现需求不清或设计有风险，先停下来指出问题。
```

### 5. 系统化调试 / 修 Bug
适合 bug 排查，不想让它直接乱改。

```text
帮我系统化排查这个问题。
先不要直接改代码，先做：
1. 复述问题和影响范围
2. 列出可能原因
3. 设计最小复现 / 验证方法
4. 定位根因
5. 给出修复方案
6. 最后再实施并验证
问题描述：
[你的 bug 描述]
```

简洁版：

```text
先做 root cause analysis，不要先盲改代码。
请先定位根因，再给修复方案，最后再改并验证。
问题是：
[问题描述]
```

### 6. 代码评审 / Review
适合让它像 reviewer 一样工作。

```text
帮我做一次 code review。
重点关注：
- bug 风险
- 行为回归
- 边界条件
- 缺失测试
- 不合理的复杂度
请先给 findings，按严重程度排序，再给简短总结。
```

如果你要 review 某次改动：

```text
帮我 review 当前改动。
重点找真正会出问题的地方，不要泛泛而谈。
先列 findings，再说总结和残余风险。
```

### 7. 收尾 / 提交前检查
适合功能做完后的收官。

```text
这个功能已经实现完了。
请帮我做上线前检查：
- 是否和原 spec 一致
- 测试是否充分
- 是否有明显回归风险
- 是否还有边界场景没覆盖
- 是否可以进入提测 / PR
请先列问题，再给结论。
```

### 8. 从 PRD 到开发的一条龙模板
这个很适合平时直接整段使用。

```text
我们按下面流程协作：

1. 先进入 PRD / spec 阶段
2. 你分块输出 spec 给我确认
3. 在我确认前，不要开发
4. spec 确认后，再单独输出 implementation plan
5. 计划确认后，再按 TDD 开发
6. 开发完成后，做一次 code review 和上线前检查

本次需求是：
[你的需求]
```

### 9. 推荐默认模板
适合“先写 PRD，放到 Google Docs，确认后再开发”的场景。

```text
进入 PRD 阶段。根据下面这次迭代需求先写一版 PRD，并准备放到 Google Docs。
要求：
- 先不要开发
- 先输出 PRD 草案给我确认
- 我确认后再写入 Google Docs
- PRD 确认完成后，再单独输出 implementation plan
- implementation plan 确认后，再按 TDD 进入开发

需求如下：
[你的需求]
```

### 10. 最短万能开场句
懒得写很多时，直接这几句也够用。

```text
先不要写代码，先把需求定义清楚。
```

```text
先出 spec，我确认后再出 implementation plan。
```

```text
按 TDD 实现，不要跳过测试。
```

```text
先做 root cause analysis，不要直接改代码。
```

```text
先做 code review，重点找风险和缺失测试。
```
