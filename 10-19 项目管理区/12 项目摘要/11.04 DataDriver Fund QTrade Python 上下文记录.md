## 2026-04-21

### 会话主题
- 分析后端项目中的缠论园 / TradingView 缠论模块，重点核查买卖点算法、季线/年线聚合逻辑，以及 TradingView 可绘制性。

### 已确认结论 / 决策
- 当前 `/tradingview/history` 实际走的是 `ChanlunTradingViewViewSet`，不是旧版 `TradingViewViewSet`，两套实现口径并不一致，评估时应以 `backend/dvadmin/selection/views/chanlun.py` 为主。
- 买卖点核心逻辑主要来自 `backend/ZS_sig_ai.py`，其中 B1/S1 采用中枢 + MACD 面积背驰，B2/S2、LB2/LS2、B3/S3 为二次规则扩展；季线/年线还额外启用独立 B2/S2 检测和纯笔背驰补点。
- 季线/年线在新接口中，如果数据库无对应 period，会从日线按 `QS` / `YS` 聚合，时间戳再统一对齐到季初/年初，整体口径比旧接口稳定。
- TradingView 历史接口当前仅返回 K 线、笔、笔中枢、买卖点；`xds`、`xd_zss`、`zsd_zss`、`bcs` 在新接口中仍为空，因此线段、线段中枢、走势段中枢目前不能通过现有 `/tradingview/history` 直接绘制。
- 返回的 `bis` / `bi_zss` / `mmds` 属于 UDF 扩展字段，不是 TradingView 原生标准字段，是否显示依赖前端自定义 overlay 逻辑，而不是后端返回后自动可画。

### 本次分析 / 涉及范围
- `backend/dvadmin/selection/views/chanlun.py`
- `backend/dvadmin/selection/views/tradingview.py`
- `backend/ZS_sig_ai.py`
- `backend/czsc/docs/check_bi_vs_chanlun.md`
- `backend/czsc/docs/check_bi_endpoint_selection.md`
- `backend/czsc/czsc/py/objects.py`

### API / 数据流影响
- 影响接口：`/selection/tradingview/history`
- 数据流：数据库 `stock_kline` -> 周期映射 / 可选日线聚合 -> `czsc.CZSC` 生成笔 -> `ZS_sig_ai` 计算中枢/买卖点 -> 扩展字段返回给前端 TradingView 容器。
- 旧版 `TradingViewViewSet` 仍保留不同聚合与返回逻辑，存在实现分叉风险。

### 验证情况
- 已静态核对新旧两个 history 实现的周期映射、聚合规则、时间戳对齐策略和输出字段。
- 已逐段审阅 `ZS_sig_ai.py` 的中枢、B1/B2/B3/S1/S2/S3、LB2/LS2、线段与笔背驰实现。
- 本次未连接数据库做样本回放，也未联动前端实际渲染验证，结论基于代码静态分析。

### 风险 / 待办
- 高周期独立 B2/S2 与纯笔背驰补点是工程化增强，不是严格标准缠论定义，存在过度出信号风险。
- B3/S3 的代码条件更接近“中枢最后一笔后的回抽/反抽”简化实现，可能与严格的突破后回踩定义存在偏差。
- 线段算法为自定义启发式实现，且未接入 `/tradingview/history` 返回，当前前端无法完整验证线段链路。
- 若要确认“准确性”，下一步需要抽具体股票和周期做逐笔对照回放，并联动前端验证 overlay 画图结果。

### 下一步建议
- 先选 1 到 2 个样本标的，对日线、季线、年线的买卖点做逐笔回放核验。
- 明确前端是否真的消费 `bis` / `bi_zss` / `mmds` / `xds` 等扩展字段，避免误以为 TradingView UDF 会原生绘制。
- 如果要支持线段和线段中枢展示，应把 `seg/xds/xd_zss` 正式接入当前 `/tradingview/history` 响应，并统一新旧实现口径。
