# 当前状态

## 摘要块
- 这篇讲什么：适合优先给人和 AI 看的项目现状快照。
- 当前结论：项目系统骨架已建立，当前主要任务是把 Python/Django 后端、量化业务链路和运行验证方式逐步沉淀成可反查卡片。
- 已过期内容：旧式单文件 Obsidian 入口不再作为唯一上下文入口。
- 下一步看什么：如果是开发任务，先去 [[00 模块地图总览]] 和 [[00 API 契约总览]]。

## 当前目标
- 建立 QTrade Python 项目长期可维护的 Obsidian 项目系统。
- 让后续协作能快速定位 Django 应用、API、服务、任务、量化算法与数据流。
- 把会话里反复出现的稳定结论逐步提升为卡片，而不是继续堆在单一日志里。

## 当前关注点
- 当前重点模块：`dvadmin/selection`、`core/chanlun`、`backend/czsc`、`application`、`dvadmin/system`。
- 当前重点链路：API 路由 -> 视图 -> service/task_service -> 缠论/选股/回测核心逻辑。
- 当前工程关注：依赖、启动脚本、Django/Celery/Redis/MySQL 环境、测试与验证命令。

## 当前有效结论
- 仓库是以 `backend` 为核心的 Python/Django 后端项目，包含系统管理、选股分析、缠论核心和 czsc 相关能力。
- `dvadmin/selection` 是量化业务高价值入口。
- `core/chanlun` 与 `backend/czsc` 是算法和分析能力高价值入口。
- `application/urls.py`、各应用 `urls.py`、views 和 services 是 API 反查主链路。
- 当前 Obsidian 骨架以参考 11.03 项目的组织方式建立，但内容应持续按本项目实际代码修正。

## 当前风险
- 项目初始上下文仍偏骨架化，细卡需要通过后续真实任务逐步补齐。
- `backend/czsc` 可能包含上游库或迁入代码，修改前需要分清本项目定制与第三方源代码边界。
- 本次未运行完整后端测试，只做了结构与文件级验证。
- 当前仓库已有未归属本次的改动：`backend/requirements.txt`、`start.sh`，后续处理时不要误回滚。

## 当前待补
- 运行方式与环境变量卡。
- 选股、回测、TradingView、行情同步 API 细卡。
- 数据库模型与任务调度关系卡。
- 缠论核心算法边界与回归保护卡。

## 下一步建议
- 每次新增会话结论时，判断它属于日志、知识、决策还是交付。
- 从高频模块开始补链路，不追求一次性全量重构。
- 优先补“模块 -> API -> 服务 -> 数据/任务”的反查链。
