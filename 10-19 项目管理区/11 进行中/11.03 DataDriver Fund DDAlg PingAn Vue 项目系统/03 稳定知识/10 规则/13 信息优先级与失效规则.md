# 13 信息优先级与失效规则

## 摘要块
- 这篇讲什么：定义项目系统里不同层级信息的优先级、冲突处理和失效规则。
- 当前结论：不能再把旧日志、旧结论和当前事实混用；读和写都必须按优先级执行。
- 已过期内容：不要默认把会话日志当成当前事实来源。

## 默认判定公式
- 默认采用：`当前状态 > 最近交付 / 最近问题 > 稳定知识 > 会话日志`。
- 如果 [[10-19 项目管理区/11 进行中/11.03 DataDriver Fund DDAlg PingAn Vue 项目系统/01 当前状态/00 当前状态]] 与稳定知识冲突，默认以前者为准。
- 如果怀疑 [[10-19 项目管理区/11 进行中/11.03 DataDriver Fund DDAlg PingAn Vue 项目系统/01 当前状态/00 当前状态]] 已过期，必须由最近交付、最近问题或最近会话记录交叉确认。

## 冲突处理
- 如果当前状态与其他卡片冲突，以 [[10-19 项目管理区/11 进行中/11.03 DataDriver Fund DDAlg PingAn Vue 项目系统/01 当前状态/00 当前状态]] 为准。
- 如果某次改动已经落到交付记录，且问题排查或稳定知识仍保留旧口径，以最新交付记录为准。
- 如果某个问题已经形成新的排查结论，旧问题结论应标记为历史参考或失效。
- 会话日志只提供时间线，不直接覆盖稳定知识或当前状态。

## 状态语义
- `active`：当前有效，可以作为现状使用。
- `temporary`：临时结论，只能作为阶段判断，后续可能反转。
- `confirmed`：问题、决策或排查结论已经确认，但仍需看是否被后续变更覆盖。
- `completed`：交付已完成，但仍要看是否被新交付替代。
- `deprecated`：已失效，只保留历史参考价值，不应继续作为当前事实。

## 升级规则
- 能脱离当天会话单独理解、且后续仍可能复用的内容，必须从日志升级到结构化卡片。
- 模块职责、页面职责、页面到 API 的固定映射，升级到模块卡 / 页面卡 / API 卡。
- “为什么这样做”的方案取舍，升级到决策卡。
- 能复用的问题现象、根因和验证方法，升级到问题排查卡。
- 已完成的改动闭环、验证结果和风险，升级到交付记录。

## 不升级规则
- 只是一时猜测、尚未验证的判断，不升级到稳定知识层。
- 纯过程性聊天、排查尝试、尚未形成结论的分支，先留在会话日志。
- 已经被新结论覆盖但还未清理的旧描述，不直接复制到新卡片。
- 未验证内容不得写成稳定事实，也不得进入“当前结论”区。

## 失效规则
- 当新的当前状态、问题结论或交付记录覆盖旧内容时，旧卡片要明确标注为历史参考或 `deprecated`。
- 任何已失效内容都不应继续出现在“当前结论”里。
- 会被替代的结论不要长期停留在当前状态；一旦失效，应移出当前状态并保留在历史卡片中。

## 失效处理动作
1. 把失效内容从原卡的“当前结论”中移走。
2. 在原卡的 frontmatter 或正文中标记为 `deprecated` 或“历史参考”。
3. 在原卡补充“被谁覆盖”的链接，明确跳转到新的问题卡、交付记录、决策卡或当前状态。
4. 如果旧内容还会误导读取顺序，就不要继续把它放在首页、总览页或当前状态的主入口区。

## 局部失效规则
- 允许整张卡 `deprecated`，也允许只把卡内某一条“当前结论”标记为失效。
- 如果只有局部结论失效，不必废弃整张卡；应把失效段落移到“历史参考”区，并补“被谁覆盖”的链接。
- 如果只是某段背景仍有参考价值，可保留为“历史参考”，但不能继续挂在“当前结论”区。
