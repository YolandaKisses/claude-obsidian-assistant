# 当前状态

## 摘要块
- 这篇讲什么：适合优先给人和 AI 看的项目现状快照。
- 当前结论：项目仍处于历史沉淀深、定位成本高的状态，优先需要稳定的模块/API 入口索引。
- 已过期内容：不再把所有当前事实混在历史日志里。
- 下一步看什么：如果是开发任务，先去 [[00 模块地图总览]] 和 [[00 API 契约总览]]。

## 当前目标
- 让项目上下文在固定骨架下持续保持可检索、可反查、可失效。
- 让后续每次协作都能快速定位到模块、页面和接口域。
- 把会话里反复出现的稳定结论逐步提升为卡片，而不是继续堆在单一日志里。

## 当前关注点
- 当前重点模块：`metricCenter`、`newReport`
- 当前聚焦页面 / 链路：指标中心高频页面定位链、`ReportRenderer` 渲染链、页面到 API 的反查链
- 当前优化重点：减少“先读日志再猜入口”的路径，优先走模块卡 / 页面卡 / API 卡
- 当前暂不处理的模块 / 页面：XBRL、控制台的低频页面，以及未进入当前问题链路的历史边角页面

## 当前有效结论
- 这个项目最适合“页面目录 + API 域”成对查找。
- `metricCenter`、`reportConfig`、`dataReport`、`newReport`、`xbrl` 是最常用的高价值入口。
- 图形化相关需求需要先从使用页面反查组件，不适合直接从组件库盲搜。
- `ReportRenderer` 的树表识别已经确认存在误判风险，普通列表仅有 `id` 时不应直接按树表处理。
- 组合树管理、`select-table`、资源管理上传白名单是近期已经有结论的高价值改动点。

## 当前风险
- 页面命名和目录命名不完全统一。
- `old`、`copy`、`backup` 等历史文件会干扰入口判断。
- 部分稳定知识仍停留在运行日志和少量总览页，尚未完全拆成细卡。
- 如果后续不持续做失效标记和链路补强，骨架仍会重新退化成“分类文件夹”。

## 当前待补
- 问题卡 -> 决策卡 -> 交付记录 的闭环继续补强
- 更多页面卡与 API 卡的双向关联
- 失效内容的 `deprecated` 标记
- 更细的页面级问题排查卡和交付记录

## 下一步建议
- 每次新增会话结论时，判断它属于日志、知识、决策还是交付。
- 从高频模块开始持续补链路，不追求一次性全量重构。
- 优先补“问题 -> 决策 -> 交付”和“模块 -> 页面 -> API”的缺口。
