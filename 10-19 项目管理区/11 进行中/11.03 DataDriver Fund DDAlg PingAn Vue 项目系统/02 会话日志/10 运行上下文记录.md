## 2026-04-22（ReportRenderer 树表误判排查）

### 会话主题
- 排查公共组件 `ReportRenderer` 在 `hsbc-report` 初始化时出现“数据格式错误”的原因。
- 核对该组件是否误把普通列表数据识别为树形表格。

### 已确认结论 / 决策
- `ReportRenderer` 当前树表判断并不只是看是否显式配置 `treeConfig`，还会在 `processTableData` 中根据返回数据字段自动识别。
- 原逻辑为“任意一行存在 `id` 或 `pid` 任一字段，就进入树表分支”，这会把仅包含普通主键 `id` 的平铺列表误判为树表。
- `common.converTreeData` 要求数据中存在可识别的根节点；一旦普通列表被误送入该方法，就会触发“数据格式错误”提示。
- 已将判断逻辑收紧为“数据中同时存在树节点 id 字段和父节点 pid 字段时，才进入树表分支”。

### 本次变更 / 涉及范围
- 从 `ReportRenderer` 中抽离树表字段解析与判断逻辑到独立工具模块。
- 修正 `processTableData` 的树表识别条件，避免普通列表误走树形转换。
- 新增最小化验证脚本，覆盖“仅有 id 不应视为树表”和“同时有 id/pid 应视为树表”两种场景。

### 涉及文件 / 模块
- `src/views/newReport/ReportRenderer.vue`
- `src/views/newReport/utils/treeTable.js`
- `tests/newReport/treeTable.spec.js`

### API / 数据流影响
- 不修改接口请求参数、接口地址或返回结构。
- 仅调整前端拿到 `dataList` 之后的本地判定逻辑。
- 若后端返回普通列表但含有业务主键 `id`，现在会按平铺表格渲染，不再尝试调用树表转换。

### 验证情况
- 已通过 `node tests/newReport/treeTable.spec.js` 验证关键判断：
  - 只有 `id` 的普通列表不会进入树表。
  - 同时具备 `id` / `pid` 的数据仍会进入树表。
- 尝试执行 `eslint` 校验目标文件时，项目现有 ESLint 在该环境下报 `Cannot read property 'range' of null`，属于工具链异常，未能据此完成 lint 通过确认。

### 风险 / 待办
- 若后续存在“树表数据字段名并非默认 `id/pid`，且部分行缺失父字段”的特殊接口，需要结合 `treeConfig.parentField / idField` 再做样本验证。
- 当前仅补了纯逻辑验证，尚未在真实页面联调里复现 `hsbc-report` 接口返回后再次确认 UI 表现。

### 下一步建议
- 用真实 `hsbc-report` 数据再走一遍初始化链路，确认页面不再弹“数据格式错误”且表格按普通列表渲染。
- 若后续准备把 `ReportRenderer` 投入更多报表场景，可继续补一组针对自定义 `treeConfig` 字段名的验证用例。
