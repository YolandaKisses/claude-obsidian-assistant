# 10 上下文读取顺序与沉淀规则

## 背景
- 项目历史沉淀深，单一笔记既难检索，也容易把过期内容当作当前事实。
- 协作中需要兼顾人读、AI 注入和长期维护。

## 决策
- 进入项目时，统一按 `首页 -> 当前状态 -> 会话日志 -> 相关知识卡 / 决策卡 / 交付记录` 的顺序读取。
- 会话日志只保留时间线和阶段性讨论，不再承担长期稳定知识库职责。
- 稳定知识拆入“稳定知识 / 模块地图 / API 契约 / 问题排查 / 决策记录 / 交付记录”。

## 硬规则
- 当前状态：只放当前仍然有效、需要优先注入上下文的事实。
- 会话日志：只放过程、讨论、阶段性判断和时间线。
- 稳定知识：只放长期有效、可复用、适合多次引用的结论。
- 问题排查：只放“现象 -> 根因 -> 验证”的问题闭环。
- 决策记录：只放“为什么这么做”的取舍依据。
- 交付记录：只放“做了什么、影响了什么、怎么验证”的结果闭环。

## 影响
- 提高上下文检索效率。
- 降低把历史讨论误当成当前事实的风险。
- 要求后续每次会话结束后判断应更新哪一层，而不是只追加日志。
